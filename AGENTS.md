# AGENTS.md

## 1. プロジェクト概要
OpenTableOrder は、日本向け飲食店の運用を支援する OSS（予約・来店・注文・会計・分析）を実装するプロジェクト。
Codex は以下を常に優先する。
- 保守性
- 型安全性
- レビューしやすさ
- 実務に近い責務分離
- 将来の拡張性（reservation analytics / business-dashboard / 実滞在時間ベース KPI）

## 2. リポジトリ構成の前提
- `apps/web`: Next.js / React / Tailwind の UI 実装（画面・hooks・components）
- `apps/api`: NestJS の API 実装（controller・service・domain ロジック）
- `packages/shared`: web / api で共有する型・定義
- `prisma`: Prisma schema・migration・seed など DB 定義

## 3. 実装ルール
- `any` は原則禁止。やむを得ない場合は理由を説明できる状態で最小範囲に限定する。
- TypeScript の型を明示し、暗黙依存を増やさない。
- Prisma schema / API レスポンス / frontend 型の整合性を常に保つ。
- 不要な大規模リファクタはしない。依頼範囲内で最小差分を優先する。
- 既存構成・既存命名・既存責務を尊重し、無関係な設計変更を持ち込まない。
- UI 文言は日本語を優先する（既存方針がある箇所はそれに合わせる）。
- 日付・時刻・timezone は既存仕様を確認し、破壊的変更をしない。
- helper / util の分割は責務が明確に分かれる場合のみ行う。
- コメントは最小限（「なぜ必要か」がコードだけで伝わらない箇所のみ）。
- 変更差分はレビューしやすく小さく保つ（無関係な整形・rename を避ける）。

## 4. 実装の進め方（毎回固定）
Codex は毎回、以下の順で作業する。
1. 変更方針を整理する
2. 変更対象ファイル一覧を出す
3. 実装する
4. 必要に応じて schema / migration / seed を更新する
5. 必要に応じて API / hooks / components / page を一貫して更新する
6. 最後に確認事項（影響範囲・未対応・確認結果）をまとめる

## 5. PR / コミットルール
- PR タイトルは簡潔な日本語で書く。
- PR 本文は日本語で書く。
- コミットメッセージは日本語で書く。
- PR 本文には必ず以下を含める。
  - 目的
  - 変更内容
  - 影響範囲
  - 動作確認
  - 未対応
- PR タイトル例:
  - business-dashboard に時間帯別客席稼働率を追加
  - TableSession の実滞在時間集計を追加
  - 予約分析画面の型定義を整理

## 6. コードレビュー観点
- 型安全性を満たしているか
- ドメイン整合性（store / table / tableSession / reservation / order / payment）が保たれているか
- 日時 / timezone の扱いが仕様どおりか
- API レスポンスとフロント型が一致しているか
- 命名規則と責務分離に一貫性があるか
- 無関係な変更が混ざっていないか
- 将来の analytics / dashboard 拡張に耐える設計か

## 7. ダッシュボード / analytics 実装時の追加ルール
- KPI 定義を曖昧にしない（指標名だけ先行させない）。
- 分母 / 分子を説明可能なデータ構造・命名にする。
- 近似値か実測値かをコードとレスポンス定義で明確にする。
- 金額は UI で読みやすく整形（桁区切り・通貨表記）する。
- グラフ追加時は、先に型定義と API レスポンス契約を揃える。
- Recharts 利用時も既存 UI トーン（色・余白・タイポグラフィ）に合わせる。

## 8. 禁止事項
- ドキュメントのみ更新して実装を行わないこと（実装タスク時）。
- 実装に不要なファイル名変更・大規模 rename。
- 無関係なフォーマット変更・lint 修正の混在。
- 根拠のない `any` の追加。
- 未確認仕様を断定して広範囲を変更すること。

## 9. AGENTS.md 自体の更新ルール
- Codex が同じミスを繰り返した場合、再発防止の追記候補を提案する。
- ただし肥大化を避け、反復発生するルールのみを追加する。
